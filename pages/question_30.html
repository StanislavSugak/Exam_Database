Участники процесса репликации: издатель

Издателем называется сервер, который предоставляет расположенные на нем данные для копирования на другие серверы. Для одного издателя может быть сконфигурировано множество подписчиков. Таким образом, одна копия данных становится доступным множеству серверов. Издатель может одновременно являться подписчиком по отношению к другому издателю.

Помимо создания копий данных издатель отслеживает вносимые пользователями в локальную базу данных изменения и в зависимости от конфигурации подготавливает новую копию. Публикуемые данные могут иметь только одного издателя, даже если они были в последствии изменены подписчиком.

В зависимости от использованного метода репликации подписчики могут или не могут вносить изменения в реплицированные данные. В простейшем случаи при репликации моментальных снимков изменять данные может только издатель. При использовании более сложных моделях репликации изменении так же могут вносить подписчики.

Изменённые данные, полученные от всех подписчиков, синхронизируются и объединяются с данными издателя, а затем снова распространяются всем участникам репликации, включая читателя.

Издатель поддерживает всю информацию обо всех сконфигурированных обо нем статьях и публикациях.

Участники процесса репликации: подписчик

Подписчиком называется сервер, копирующий предоставляемые издателем данные. В качестве подписчика может выступать не только SQL Server, но и SQL Server предыдущих версий, а также другие СУБД. Подписчик может быть сконфигурирован в качестве издателя для других подписчиков.

Механизмы репликации SQL Server разрешает вносить изменения не только издателям, но и подписчикам. Механизмы, использующие изменения данных подписчиком, отличается от тех, которые используется при изменениях данных издателем.

Существуют два методы обновления подписчиков:

1) Pull subscription (репликация по запросу). При использовании этого метода подписчик периодически подключается к дистрибьютору и запрашивает у него все изменения, сделанные со времени последнего подключения. При наличии большого количество подписчиков использование репликации по запросу позволяется снизить нагрузку на дистрибьютора. Кроме того, для мобильных пользователей также рекомендуется использовать репликацию по запросам. В этом случае, они могут начать процесс репликации немедленно после подключения к сети, а не дожидаться, пока дистрибьютор скопирует им данные. Использование репликации по запросу позволяется упросить процесс рассылки данных через глобальные сети, например через интернет, при этом подписчик инициирует начало репликации наиболее удобное для него время

2) Push subscription (принудительная репликация). Дистрибьютор сам устанавливает соединение с подписчиками и копирует им все необходимые данные. Использование этого метода репликации рекомендуется для сервера с которыми постоянно установлено соединение. Изменения могут копироваться постоянно сразу после того, как они произошли, или периодически на основе установленного расписания. Администратор может централизованно управлять расписанием обновлений на всех подписчиков

Участники процесса репликации: дистрибьютор

Дистрибьютор называется сервер, поддерживающий базу данных распределения. Этот сервер используется в роли посредника между издателем и подписчиком. Дистрибьютор копирует себе все публикации, подготовленные издателем, и тиражирует их подписчикам. Использование дистрибьютора позволяется разгрузить основной сервер, переложив заботу об отслеживании процесса репликации на специально выделенного для этого сервер.

Кроме того, в некоторых случаях дистрибьютор может значительно увеличить скорость выполнения операций тиражирования данных за счет снижения сетевого трафика. В качестве дистрибьютора можно использовать как выделенный сервер, так и сервер, сконфигурированный в качестве издателя или подписчика. Конкретные функции, выполняемые дистрибьютером, зависят от использованного метода репликации.
